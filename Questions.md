# Questions

## 2026-06-25

TasteFilter 백엔드를 만들면서 헷갈렸거나 확인했던 내용을 질문/답변 형식으로 정리한 문서입니다.

## 단계별 진행 정리

### Phase 1. 프로젝트 기반 세팅

무엇을 했나:

* Docker 기반 PostgreSQL 환경을 준비했습니다.
* `.env`로 DB 접속 정보와 네이버 API 키를 분리했습니다.
* Spring Boot가 환경 변수를 읽을 수 있도록 `application.yaml`을 구성했습니다.

왜 했나:

* 민감한 값을 코드에 직접 넣지 않기 위해서입니다.
* 로컬 개발 환경에서도 DB와 API 설정을 안정적으로 관리하기 위해서입니다.

### Phase 2. 도메인 모델 구성

무엇을 했나:

* `Restaurant` Entity를 만들었습니다.
* `Review` Entity를 만들었습니다.
* `Region`, `FoodCategory` Enum을 만들었습니다.

왜 했나:

* 식당 정보와 리뷰 정보를 1:N 구조로 분리하기 위해서입니다.
* 리뷰만 계속 쌓이고 식당 정보가 중복 저장되는 구조를 피하기 위해서입니다.

### Phase 3. Repository 구성

무엇을 했나:

* `RestaurantRepository`를 만들었습니다.
* `ReviewRepository`를 만들었습니다.
* 네이버 플레이스 ID, 블로그 URL, 식당별 리뷰 조회 메서드를 추가했습니다.

왜 했나:

* Service가 DB 접근 코드를 직접 들고 있지 않게 하기 위해서입니다.
* Spring Data JPA의 쿼리 메서드로 반복적인 조회 로직을 단순화하기 위해서입니다.

### Phase 4. Service 계층 구성

무엇을 했나:

* `RestaurantService`를 만들었습니다.
* `ReviewService`를 만들었습니다.
* `ReviewFilterService`를 만들었습니다.
* 광고/협찬/체험단/리뷰 이벤트 문구를 기준으로 리뷰 신뢰도를 계산하도록 했습니다.

왜 했나:

* Repository 호출을 Controller나 Crawler에 흩뿌리지 않기 위해서입니다.
* 리뷰 저장 전에 반드시 필터링 규칙을 거치도록 만들기 위해서입니다.

### Phase 5. 네이버 블로그 검색 API 연동

무엇을 했나:

* 네이버 개발자센터에서 검색 API 애플리케이션을 등록했습니다.
* `NAVER_CLIENT_ID`, `NAVER_CLIENT_SECRET` 환경 변수를 추가했습니다.
* `NaverBlogSearchClient`를 만들어 네이버 블로그 검색 API를 호출하도록 했습니다.
* 검색 결과 DTO를 만들었습니다.

왜 했나:

* 무작정 웹을 긁는 것이 아니라 공식 검색 API로 후보 블로그 글을 먼저 찾기 위해서입니다.
* API 키를 코드가 아니라 환경 변수로 관리하기 위해서입니다.

### Phase 6. 검색 후보 변환

무엇을 했나:

* `SearchQueryFactory`를 만들었습니다.
* 지역과 음식 카테고리로 `성수 한식 맛집` 같은 검색어를 만들도록 했습니다.
* `NaverBlogCandidateService`를 만들어 검색 결과를 `CrawledReview` 후보로 변환했습니다.
* `CrawlCandidateSearchResult` DTO를 추가했습니다.

왜 했나:

* 네이버 검색 API 결과는 최종 리뷰가 아니라 상세 분석이 필요한 후보이기 때문입니다.
* 검색 결과를 바로 DB에 저장하지 않고, 다음 파싱 단계로 넘기기 위해서입니다.

### Phase 7. 상세 본문 파싱

무엇을 했나:

* Jsoup 의존성을 추가했습니다.
* `NaverBlogPostParser`를 만들었습니다.
* 네이버 블로그의 `mainFrame`, `se-main-container`, `postViewArea` 구조를 분석했습니다.
* 검색 API 요약이 아니라 실제 블로그 본문을 가져오도록 했습니다.

왜 했나:

* 네이버 검색 API만으로는 광고 문구, 공정위 배너, 영수증 인증 여부를 충분히 판단할 수 없기 때문입니다.
* 정확한 필터링을 하려면 본문과 상세 DOM 구조가 필요하기 때문입니다.

### Phase 8. 크롤링 오케스트레이션

무엇을 했나:

* `NaverBlogCrawlOrchestrator`를 만들었습니다.
* 식당 조회, 후보 검색, 중복 URL 확인, 상세 파싱, 필터링, 저장을 한 흐름으로 묶었습니다.
* `CrawlIngestionResult` DTO를 추가해 실행 결과를 집계하도록 했습니다.

왜 했나:

* Controller나 Scheduler가 여러 Service를 직접 순서대로 호출하지 않게 하기 위해서입니다.
* 수동 실행 API와 스케줄러가 같은 수집 로직을 재사용할 수 있게 하기 위해서입니다.

### Phase 9. 문서화

무엇을 했나:

* 네이버 블로그 검색 API 문서를 `real_docs`에 정리했습니다.
* README에 2026-06-22 네이버 API 연결 기록을 남겼습니다.
* README에 2026-06-25 크롤링 파이프라인 작업 기록을 남겼습니다.
* `Questions.md`에 헷갈렸던 개념과 작업 흐름을 정리했습니다.

왜 했나:

* 나중에 어떤 판단으로 구조를 만들었는지 다시 확인하기 위해서입니다.
* 학습 중 헷갈린 내용을 프로젝트 안에 남겨두기 위해서입니다.

### 다음 Phase 후보

가장 자연스러운 다음 순서는 수동 실행 API입니다.

1. 관리자용 수동 크롤링 API를 만듭니다.
2. 특정 식당 ID로 `NaverBlogCrawlOrchestrator`를 호출합니다.
3. 응답으로 후보 수, 저장 수, 실패 수를 확인합니다.
4. IntelliJ나 Postman으로 실제 DB 저장까지 확인합니다.
5. 흐름이 안정화되면 Scheduler를 붙입니다.

이 순서가 좋은 이유는 자동 스케줄러를 먼저 붙이면 실패했을 때 원인을 찾기 어렵기 때문입니다.
먼저 수동으로 한 번씩 실행하면서 검색, 파싱, 필터링, 저장이 제대로 이어지는지 확인하는 편이 안전합니다.

### 1. 크롤링은 어떻게 처리하려고 해?

유저가 요청할 때마다 실시간으로 외부 사이트를 긁는 방식은 피합니다.
대신 백그라운드 스케줄러나 관리자 실행 API를 통해 미리 데이터를 수집하고, 필터링한 뒤 자체 DB에 저장합니다.

이렇게 하는 이유는 다음과 같습니다.

* API 응답 속도를 안정적으로 유지할 수 있습니다.
* 외부 사이트에 순간적으로 많은 요청을 보내는 일을 줄일 수 있습니다.
* IP 차단이나 요청 제한 위험을 낮출 수 있습니다.
* 이미 필터링된 리뷰만 DB에서 빠르게 조회할 수 있습니다.

현재 흐름은 네이버 블로그 검색 API로 후보 글을 찾고, Jsoup으로 상세 본문을 파싱한 뒤, 광고/협찬/이벤트 필터를 거쳐 저장하는 방향입니다.

### 2. Repository는 왜 만들었어?

Repository는 DB에 접근하는 계층입니다.

`RestaurantRepository`는 식당 데이터를 조회하거나 저장할 때 사용합니다.
특히 `naverPlaceId` 기준으로 이미 등록된 식당인지 확인하기 위해 만들었습니다.

`ReviewRepository`는 리뷰 데이터를 조회하거나 저장할 때 사용합니다.
특정 식당의 리뷰 목록을 최신순으로 가져오고, 리뷰 개수를 세고, 같은 블로그 URL이 이미 저장됐는지 확인합니다.

Repository를 따로 두면 Service가 SQL이나 EntityManager를 직접 다루지 않아도 되고, Spring Data JPA가 메서드 이름만 보고 쿼리를 만들어줍니다.

### 3. 테스트는 아예 안 돼?

테스트 자체는 가능합니다.
다만 현재 프로젝트는 PostgreSQL, 환경 변수, 네이버 API 키, 외부 네트워크 요청 같은 조건이 얽혀 있어서 테스트 종류를 나눠야 합니다.

가볍게 가능한 테스트는 다음과 같습니다.

* 필터링 로직 단위 테스트
* 검색어 생성 로직 테스트
* DTO 생성/검증 테스트

조금 준비가 필요한 테스트는 다음과 같습니다.

* Repository 테스트
* 실제 PostgreSQL 연결 테스트
* 네이버 API 연동 테스트
* Jsoup 상세 페이지 파싱 테스트

그래서 당장은 코드를 먼저 구성하고, 테스트는 사용자가 IntelliJ에서 직접 실행해보는 방식으로 진행했습니다.

### 4. JPA 세팅은 되어 있는데 Entity가 DB에 생성이 안 되는 이유는?

가능성이 큰 원인은 `spring.jpa.hibernate.ddl-auto` 설정입니다.

`ddl-auto`가 `none`이나 `validate`면 Entity를 보고 테이블을 자동 생성하지 않습니다.
개발 단계에서 자동 생성을 원하면 보통 `update` 또는 `create`를 사용합니다.

다만 `create`는 애플리케이션 실행 시 기존 테이블을 지울 수 있으므로 조심해야 합니다.
개발 중에는 `update`, 운영 환경에서는 Flyway 같은 마이그레이션 도구를 사용하는 방식이 더 안전합니다.

### 5. Service에는 어떤 메서드를 구현하려고 했어?

Service는 Repository를 직접 호출하는 곳이 아니라, 비즈니스 규칙을 담는 계층으로 잡았습니다.

`RestaurantService`에는 다음 역할을 넣었습니다.

* 식당 ID로 식당 조회
* 네이버 플레이스 ID 기준으로 기존 식당 재사용
* 없으면 새 식당 생성

`ReviewService`에는 다음 역할을 넣었습니다.

* 크롤링된 리뷰를 필터링한 뒤 저장
* 이미 저장된 블로그 URL이면 중복 저장 방지
* 특정 식당의 최신 리뷰 조회
* 특정 식당의 리뷰 개수 조회

이렇게 나눈 이유는 Controller, Scheduler, Crawler가 Repository를 직접 만지지 않고 Service를 통해 같은 규칙을 재사용하게 하기 위해서입니다.

### 6. Service에 왜 주석을 남겼어?

이 프로젝트는 학습하면서 구조를 잡고 있기 때문에, 단순히 코드가 "무엇을 하는지"보다 "왜 이 메서드가 필요한지"를 남기는 게 중요했습니다.

예를 들어 중복 URL 확인 메서드는 단순 조회처럼 보이지만, 실제로는 상세 페이지 파싱 전에 불필요한 네트워크 요청을 줄이는 역할을 합니다.

그래서 주요 Service 메서드에는 다음 기준으로 짧은 주석을 남겼습니다.

* 이 메서드가 어떤 흐름에서 쓰이는지
* 왜 Repository를 바로 호출하지 않고 Service로 감쌌는지
* 성능이나 중복 방지에 어떤 의미가 있는지

### 7. 네이버 블로그 검색 API만으로 필터링할 수 있어?

완전한 필터링은 어렵습니다.

네이버 블로그 검색 API는 제목, 링크, 요약, 블로거명, 작성일 정도만 제공합니다.
본문 전체, 하단 공정위 문구, 이미지 배너, 영수증 인증 여부 같은 정보는 제공하지 않습니다.

그래서 검색 API 결과만으로 가능한 것은 1차 후보 선별 정도입니다.
정확한 광고 필터링을 하려면 블로그 상세 페이지에 들어가 본문과 DOM 구조를 추가로 분석해야 합니다.

현재 구조도 이 한계를 반영해서 검색 API 결과를 바로 저장하지 않고, `CrawledReview` 후보로 만든 뒤 상세 파싱 단계로 넘깁니다.

### 8. 네이버 API 등록할 때 뭐 넣어야 해?

네이버 개발자센터 애플리케이션 등록에서는 다음처럼 넣으면 됩니다.

* 애플리케이션 이름: 프로젝트를 구분할 수 있는 이름
* 사용 API: 검색
* 서비스 환경: 웹 서비스 URL

로컬 개발 중이면 웹 서비스 URL에는 보통 `http://localhost:8080` 또는 프론트 개발 서버 주소를 넣습니다.

검색 API만 사용할 경우에는 네이버 로그인 콜백 URL처럼 OAuth 인증에 필요한 설정은 필수가 아닙니다.
이 프로젝트에서는 `X-Naver-Client-Id`, `X-Naver-Client-Secret` 헤더로 호출하는 검색 API를 사용합니다.

### 9. `.env`에 넣은 네이버 API 키가 코드에서 읽혀?

코드에서는 `application.yaml`에 다음 형태로 연결했습니다.

```yaml
naver:
  api:
    client-id: ${NAVER_CLIENT_ID:}
    client-secret: ${NAVER_CLIENT_SECRET:}
```

즉 실행 환경에 `NAVER_CLIENT_ID`, `NAVER_CLIENT_SECRET`이 들어 있으면 Spring이 값을 주입합니다.

주의할 점은 `.env` 파일에 적었다고 해서 Spring Boot가 항상 자동으로 읽는 것은 아니라는 점입니다.
IntelliJ 실행 설정이나 Docker Compose에서 해당 값을 실제 환경 변수로 넘겨줘야 합니다.

### 10. 네이버 API 연결은 어떻게 확인했어?

`.env`에 들어 있는 `NAVER_CLIENT_ID`, `NAVER_CLIENT_SECRET` 값을 사용해서 네이버 블로그 검색 API에 실제 요청을 보냈고, HTTP 200 응답을 확인했습니다.

확인한 검색어는 `성수 한식 맛집`이었습니다.

이 작업은 README의 `2026-06-22` 개발 기록에도 남겼습니다.

### 11. 오케스트레이션이 뭐하는 거야?

오케스트레이션은 여러 개의 작은 기능을 실제 업무 흐름 순서대로 묶어서 실행하는 계층입니다.

TasteFilter 백엔드에서는 네이버 블로그 데이터를 수집할 때 다음 기능들이 각각 따로 존재합니다.

* 네이버 블로그 검색 API로 후보 글을 찾는 기능
* 이미 저장된 블로그 URL인지 확인하는 기능
* 블로그 상세 페이지에 들어가 본문을 파싱하는 기능
* 협찬, 광고, 체험단, 리뷰 이벤트 문구를 필터링하는 기능
* 필터를 통과한 리뷰만 DB에 저장하는 기능

`NaverBlogCrawlOrchestrator`는 이 기능들을 한 번에 이어 붙여 실행합니다.

실행 흐름은 다음과 같습니다.

1. 식당 ID로 DB에서 식당 정보를 조회합니다.
2. 식당의 지역과 음식 카테고리로 검색어를 만듭니다.
3. 네이버 블로그 검색 API에서 후보 글 목록을 가져옵니다.
4. 이미 저장된 블로그 URL이면 건너뜁니다.
5. 새 글이면 Jsoup으로 상세 본문을 파싱합니다.
6. 본문을 광고/이벤트 필터에 통과시킵니다.
7. 통과한 리뷰만 DB에 저장합니다.
8. 후보 수, 중복 수, 파싱 성공 수, 저장 수, 실패 수를 결과로 반환합니다.

오케스트레이션이 필요한 이유는 각 기능을 Controller나 Scheduler에 흩뿌리지 않기 위해서입니다.
흐름을 한 곳에 모아두면 나중에 스케줄러, 관리자 API, 수동 실행 API가 모두 같은 수집 로직을 재사용할 수 있습니다.

쉽게 말하면, 각 서비스가 재료라면 오케스트레이터는 조리 순서를 알고 있는 실행 담당입니다.

### 12. 다음에는 뭘 해야 해?

현재까지는 데이터 모델, Repository, Service, 네이버 검색 API 클라이언트, 상세 파서, 필터링, 오케스트레이터까지 연결했습니다.

다음 단계 후보는 다음과 같습니다.

* 관리자용 수동 크롤링 API 만들기
* 스케줄러를 붙여 주기적으로 크롤링 실행하기
* 저장된 리뷰를 조회하는 사용자 API 만들기
* 필터링 로직 단위 테스트 추가하기
* 네이버 검색 결과와 실제 식당 매칭 정확도를 높이기

지금 가장 자연스러운 다음 단계는 수동 실행 API입니다.
먼저 버튼처럼 직접 호출해서 크롤링 흐름이 DB까지 이어지는지 확인한 다음, 안정화되면 스케줄러로 자동화하는 순서가 좋습니다.

## 2026-06-27 추가 기록

### Phase 10. 관리자용 수동 크롤링 API

무엇을 했나:

* `AdminCrawlController`를 만들었습니다.
* `POST /api/admin/crawls/restaurants/{restaurantId}`로 식당 한 건의 수집 흐름을 실행하도록 했습니다.
* 검색어에 식당명을 포함하도록 보완했습니다.
* 존재하지 않는 식당 ID는 `404 Not Found`로 응답하도록 처리했습니다.

왜 했나:

* Scheduler로 자동화하기 전에 검색, 파싱, 필터링, 저장 결과를 한 번씩 직접 확인하기 위해서입니다.
* 지역과 카테고리만 검색하면 다른 식당 리뷰가 현재 식당에 저장될 수 있기 때문입니다.

### 13. 수동 크롤링 API는 왜 필요해?

Scheduler를 먼저 실행하면 검색, 파싱, 필터링, 저장 중 어디에서 실패했는지 확인하기 어렵습니다.
그래서 식당 ID 하나를 직접 지정해 전체 수집 흐름과 결과 건수를 확인하는 API를 먼저 만들었습니다.

현재는 관리자 인증이 없으므로 로컬 개발 환경에서 검증하는 용도로 사용합니다.

## 2026-06-28 추가 기록

### Phase 11. 식당별 리뷰 조회 API

무엇을 했나:

* `GET /api/restaurants/{restaurantId}/reviews` API를 만들었습니다.
* 최신 크롤링 리뷰부터 `page`, `size` 단위로 조회하도록 했습니다.
* JPA Entity 대신 응답 전용 DTO를 반환하도록 했습니다.

왜 했나:

* 수동 크롤링 후 실제 저장 결과를 API로 바로 확인하기 위해서입니다.
* 리뷰가 계속 쌓여도 전체 데이터를 한 번에 조회하지 않기 위해서입니다.
* DB 구조 변경이 API 응답 구조에 직접 영향을 주지 않도록 분리하기 위해서입니다.

### 14. 리뷰 조회 API에 페이지 처리가 왜 필요해?

리뷰는 크롤링할 때마다 계속 늘어납니다.
전체 리뷰를 한 번에 조회하면 DB 조회량, 서버 메모리 사용량, 응답 크기가 함께 커질 수 있습니다.

그래서 `page`와 `size`로 필요한 구간만 최신순으로 조회하고, 한 번에 요청할 수 있는 최대 크기를 100건으로 제한했습니다.
